인원 1 방장 + 최대 3 시청자 장르 실시간 추격 · 잠입 스택 Firebase RTDB + GitHub Pages 플랫폼 SOOP
01 — GAME FLOW

방 생성부터 정산까지

한 라운드는 방장이 만든 방에서 시작해 정산 영수증이 출력되며 끝난다. 방장은 정산 화면에서 [다시하기]로 같은 방에서 새 라운드를 반복할 수 있다.

1

방 생성

방장(스트리머)이 방을 개설. 최대 3명의 시청자가 입장 가능.

2

입장 & 대기

시청자는 도둑으로 자동 배정. 방장은 시작 버튼을 눌러 라운드를 개시.

3

카운트다운

짧은 준비 시간 후 마트 조명이 켜지며 라운드 시작.

4

정산

전원 탈출/체포 또는 제한시간 종료 시 점수 계산 후 순위 발표. 방장이 [다시하기]를 누르면 물건·위치·점수가 초기화된 채 3번(카운트다운)으로 되돌아간다.

02 — ROLES

두 개의 시점, 하나의 마트

주인과 도둑은 각자 독립된 시야를 가지며, 마트 전체는 이 시야보다 넓다 — 정보의 불균형이 아니라 위치 싸움과 시야 관리가 승부를 가른다.

방장 · 1인

마트 주인

  • 목표 — 도둑이 물건을 갖고 나가지 못하게 막는다.
  • 이동 — 도둑과 동일한 기본 속도, 무게 페널티 없음.
  • 시야 — 도둑과 동일한 크기의 개별 시야. 카메라는 캐릭터에 고정, 마우스로 시야 방향을 회전(CCTV 없음).
  • 체포 — 도둑에게 근접 후 스페이스바로 즉시 체포 판정.
시청자 · 최대 3인

도둑

  • 목표 — 물건을 최대한 많이 챙겨 출구로 탈출한다.
  • 이동 — 소지 무게가 늘수록 이동속도 감소.
  • 시야 — 주인과 동일한 크기의 개별 시야. 도둑이 여러 명이어도 서로 시야를 공유하지 않고 각자 독립적으로 본다. 카메라는 캐릭터에 고정, 마우스로 시야 방향을 회전.
  • 줍기 — 물건 위치에 닿으면 자동으로 소지품에 추가(별도 키 불필요).
  • 선택 — 적게 훔치고 빨리 튈지, 욕심내다 잡힐지 매 순간 판단.
  • 탈락 — 체포되면 해당 라운드 즉시 탈락.
03 — CORE MECHANICS

핵심 시스템 5가지

확정된 규칙과 아직 수치 조율이 필요한 항목을 구분해두었다.

SYS-01이동

무게-속도 트레이드오프

소지 무게 한도는 없음. 속도(%) = 100 − 6×무게, 최소 70%로 하한. 무게 5 이상부터는 70%로 고정되어, 아무리 많이 들어도 이동 자체가 막히지는 않는다.

확정
SYS-02체포

근접 + 상호작용 체포

주인이 도둑에게 붙어 스페이스바를 누르면 즉시 체포. 단, 대상이 훔친 물건이 없는 상태(빈손 또는 은닉 후)라면 체포가 무효 처리되고 "사과" 후 그대로 진행된다.

확정
SYS-03시야

대칭 로컬 시야

주인과 도둑(최대 3인) 모두 각자 독립된 개별 시야를 가지며, 서로 시야를 공유하지 않는다. 마우스로 시야 방향을 회전할 수 있고, 마트 전체 맵은 시야 범위보다 넓어 화면 밖 구역은 보이지 않는다. 카메라는 각 플레이어 캐릭터에 고정된다.

확정
SYS-04소음

무게 기반 소음 반경

무거운 상태로 이동하면 소음 반경이 커져 주인에게 대략적 방향을 노출한다. 시야가 대칭인 만큼 위치를 완전히 숨기지 못하게 하는 견제 장치.

채택 여부 확인 필요
SYS-05출구

다중 출구 & 차별화

정문 2개(넓고 빠르지만 노출) + 비상구/후문 2개(좁고 은신 유리)를 맵 대각선 방향으로 대칭 배치한다.

확정
SYS-06종료

라운드 종료 조건

도둑 전원이 탈출하거나 체포되면 즉시 종료. 그 전까지는 제한시간 도달 시 자동 종료.

확정
04 — SETTLEMENT

라운드 종료, 영수증이 출력된다

라운드가 끝나면 각 도둑의 결과가 정산되어 순위가 매겨진다.

MIDNIGHT MART

ROUND SETTLEMENT · #0427
도둑A — 탈출 성공+320
도둑B — 체포됨+0
도둑C — 탈출 성공+180
1위도둑A
THANK YOU · COME AGAIN

재시작 — 이 화면에서 방장에게만 [다시하기] 버튼이 노출된다. 누르면 물건·플레이어 위치·점수가 모두 초기화된 채 같은 방에서 카운트다운부터 새 라운드가 시작된다(역할은 유지).

기본안: 탈출에 성공한 도둑만 소지 물건의 가치 합을 점수로 획득한다. 체포된 도둑은 소지품을 마트에 반납(0점 처리)한다. 소지 무게 한도는 없으며, 무게가 늘수록 SYS-01 속도 공식(속도% = 100 − 6×무게, 최소 70%)에 따라 이동속도가 줄어드는 것으로 자연 제어한다.

가치와 무게는 비례하되, 상위 티어일수록 무게당 가치 효율이 소폭 높아 "고위험 고효율" 아이템 노리기가 성립한다.
스폰 개수 — 라운드 시작 시 맵 전체에 총 20개 생성, 리스폰 없음(주울수록 맵이 비어감).
티어 비율 — 피라미드형: 소 12개(60%) · 중 6개(30%) · 대 2개(10%).

06 — DATA SCHEMA

Firebase RTDB 구조

방 코드 기반 세션 + 이동 있을 때만 쓰는 절약형 동기화로 설계한다. 인원이 적어도(방당 최대 4명) 동시 방 수가 늘면 비용이 쌓이므로, 쓰기 빈도·페이로드 크기·방 정리까지 함께 절약한다.

rooms/{roomId} code: "AB12CD" // 시청자가 입력하는 6자리 방 코드 hostSessionId: string phase: "waiting" | "countdown" | "playing" | "settled" // settled → (host가 [다시하기]) → countdown 으로 순환 가능 timer: { startedAt, durationSec } ownerSpawnIndex: number // 랜덤 선택된 스폰 후보 index rooms/{roomId}/players/{sessionId} nickname: string role: "owner" | "thief" x, y: number // 정지 시 마지막 값 유지, 이동 중에만 갱신 carryWeight: number // carriedItems로부터 파생, 쓰기는 캐시용 carriedItems: { [itemId]: true } status: "active" | "arrested" | "escaped" connected: bool // onDisconnect()로 자동 false 처리 // viewAngle(마우스 시야 방향)은 서버에 없음 — 본인 로컬 렌더링 전용 상태 rooms/{roomId}/items/{itemId} x, y: number tier: "S" | "M" | "L" value, weight: number state: "onFloor" | "carried" | "escaped" | "returned" carriedBy: sessionId | null rooms/{roomId}/settlement/{sessionId} result: "escaped" | "arrested" score: number rooms/{roomId}/emotes/{sessionId} emoteId: 1 | 2 | 3 | 4 ts: serverTimestamp // 클라이언트가 2~3초 지나면 알아서 무시(별도 삭제 없음)
세션확정

방 코드 + 로컬 세션 ID

Firebase 계정 인증 없이, 방 코드로 입장하고 브라우저 localStorage에 저장된 임시 세션 ID로 플레이어를 식별한다. 새로고침해도 같은 플레이어로 복귀.

동기화확정

이동 중에만 위치 전송

이동키를 누르는 동안만 x/y를 주기적으로 쓰고, 정지 시 전송을 멈춘다. 최대 초당 전송 횟수를 제한(throttle)해 비용과 지연을 통제.

렌더링확정

다른 플레이어는 보간해서 그린다

RTDB 수신값을 그대로 스냅하지 않고, 내 화면에서는 항상 마지막 수신 위치 → 새 수신 위치 사이를 매 프레임 보간(lerp)해 이동시킨다. 전송 주기보다 렌더링 프레임이 빠르므로, 네트워크 틱 간격이 벌어져도 끊김 없이 부드럽게 보인다. 단, 내 캐릭터 본인은 입력을 즉시 반영(로컬 예측)하고 서버값은 보정용으로만 사용한다.

절약확정

시야각은 아예 전송하지 않는다

마우스 시야 방향은 체포(SYS-02, 근접+상호작용) 등 어떤 게임 로직에도 쓰이지 않고 본인 화면의 포그오브워 계산에만 필요하므로, 서버에 보내지 않고 100% 로컬 상태로만 둔다. 다른 플레이어에게 보이는 "바라보는 방향"이 필요하면 이미 전송 중인 이동 방향(dx, dy)에서 유도해 별도 필드 없이 처리한다.

절약확정

좌표는 정수로 반올림해 전송

x/y 모두 소수점 없이 정수(또는 소수 1자리)로 반올림해 페이로드 크기를 줄인다. 시야가 560×400px 수준이라 정수 단위 오차는 체감되지 않는다.

정리확정

방장이 나갈 때만 방 삭제

정산 화면에서는 자동 삭제하지 않는다 — 방장이 [다시하기]로 같은 방에서 새 라운드를 반복할 수 있기 때문이다. 방장이 명시적으로 [방 나가기]를 누르거나 onDisconnect()로 접속이 끊길 때만 remove(room)을 호출해 정리한다.

07 — UI WIREFRAME

화면엔 지표만 남긴다

미니맵 없이 시야만으로 위치를 판단하게 해 긴장감을 유지한다. HUD는 역할별로 꼭 필요한 수치만 노출한다.

OWNER HUD — 화면 좌상단

┌─────────────────────────┐ │ TIME 02:14 │ │ THIEVES LEFT 2 / 3 │ └─────────────────────────┘

THIEF HUD — 화면 좌상단

┌─────────────────────────┐ │ [█████░░░░░] 무게 7 │ │ SPEED 82% TIME 02:14 │ └─────────────────────────┘

이벤트 토스트 — 화면 상단 중앙, 2~3초 노출 후 소멸

╔═══════════════════╗ ║ 도둑B 체포됨 ║ ╚═══════════════════╝
HUD확정

미니맵 없음

전체 맵 축소도를 주지 않아 "위치 감각을 오직 시야로만 파악"하는 코어 긴장감을 지킨다. 맵을 외우거나 감을 잡는 것도 실력의 일부.

HUD확정

도둑 소지품 = 무게 막대 + 속도%

무게가 찰수록 차는 막대와 함께 현재 속도%를 숫자로 노출해, SYS-01 공식(100−6×무게, 최소 70%)에 따른 페널티를 직관적으로 체감하게 한다.

HUD확정

주인 = 남은 시간 + 남은 도둑 수

도둑과 달리 무게 페널티가 없어 소지품 UI가 필요 없다. 대신 "몇 명 더 잡아야 끝나는지"를 항상 보여줘 추격의 목표를 명확히 한다.

HUD확정

체포/탈출은 상단 토스트

순간 이벤트는 화면 전체를 가리는 연출 대신 화면 상단 중앙에 짧은 토스트 메시지로만 알린다. 시야를 가리지 않아 스트리밍 화면에서도 깔끔하다.

08 — CONTROLS

키 입력과 상호작용

이동·상호작용·이모지, 세 가지 입력만으로 조작이 끝난다. 마우스는 시야 방향 전환 전용(SYS-03).

MOVE W A S D / ↑ ← ↓ → // 동시입력 시 대각선(8방향) LOOK Mouse Move // 카메라 고정, 시야 방향만 회전 INTERACT Space // 주인: 근접 도둑 체포 / 도둑: 없음(줍기는 자동) EMOTE 1 2 3 4 // 프리셋 이모지 즉시 표시
INPUT-01이동

WASD + 방향키 동시 지원

둘 중 편한 키로 이동. 대각선 동시 입력 시 8방향으로 자연스럽게 이동한다.

확정
INPUT-02상호작용

스페이스바 = 체포 (주인 전용)

도둑에게는 상호작용 키가 없다 — 물건은 닿으면 자동으로 줍는다(줍기 위한 입력 불필요). 스페이스바는 오직 주인의 체포 판정(SYS-02)에만 쓰인다.

확정

INPUT-04이모지 도발

도둑의 이모지는 주인에게 무조건 보인다

도둑이 쓰는 이모지는 술래(주인)를 도발하기 위한 용도이므로, SYS-03 시야 범위와 무관하게 주인 화면에 항상 표시된다. 이모지를 쓴 도둑이 주인의 카메라 안에 있으면 캐릭터 머리 위에 그대로 뜨고, 화면 밖에 있으면 그 방향에 해당하는 주인 화면 테두리 끝에 이모지 아이콘이 표시된다. 도둑끼리는 이 예외가 적용되지 않고 평소처럼 서로의 시야 범위 안에서만 보인다.
